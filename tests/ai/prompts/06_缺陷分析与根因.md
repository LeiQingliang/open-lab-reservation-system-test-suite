# 提示词 06 · 缺陷分析与根因

> 用 LLM 对**人工执行中观察到的现象**做根因假设、归类与修复建议。AI 给假设，人工核实定性。

| 项目 | 内容 |
|------|------|
| **目的** | 把测试现象整理成结构化缺陷条目，给出根因假设与修复方向 |
| **适用阶段** | 流水线 ⑦ 缺陷分析 |
| **使用者** | 测试工程师（人工核实根因、定级、定编号） |

## 输入（占位符）

- `{{现象描述}}`：人工执行观察（请求/输入/实际响应/预期）
- `{{相关源码}}`：可能相关的 Controller/Service/Mapper 片段
- `{{缺陷表字段}}`：默认 编号｜标题｜模块｜严重等级｜复现步骤｜实际结果｜预期结果｜根因假设｜修复建议
- `{{已知缺陷上下文}}`：如 BUG-001 预约无冲突检测、timeSlot/日期缺校验、未授权返回 500
  （均为**历史现象示例**；BUG-001/002/003/004 当前被测源码已修复，使用时请区分"历史 vs 现状"，勿假定仍可复现）

## 提示词正文

```
你是一名缺陷分析专家。请基于"人工执行观察到的现象"和相关源码，做根因分析。

【现象】
{{现象描述}}
【相关源码】
{{相关源码}}
【输出表字段】{{缺陷表字段}}
【已知上下文】{{已知缺陷上下文}}

要求：
1. 仅对我提供的现象做分析，不要凭空新增"你觉得可能存在"的缺陷。
2. 根因假设必须落到源码（指出缺哪段校验/哪个分支没处理），并显式标注为"假设，待人工确认"。
3. 给出严重等级建议（致命/严重/一般/轻微）及理由，最终等级由人工裁定。
4. 给出最小修复方向（如在哪加 @Valid / 在 Service 加冲突检测 / 统一异常返回 401），
   但不要声称"已修复"。
5. 用 Markdown 表格输出缺陷条目，并对每条标注"复现所需前置数据"。

诚信约束：不要编造缺陷数量或"AI 自动发现了 N 个缺陷"；本分析是对人工现象的辅助归因。
```

## 期望输出

- 结构化缺陷条目表（含根因假设、修复方向、严重等级建议）；
- 每条根因标注"假设，待人工确认"与源码出处。

## 人工评审要点

- [ ] 每条缺陷对应真实观察到的现象，无凭空新增。
- [ ] 根因有源码佐证，且标注为"假设/待确认"。
- [ ] 严重等级由人工最终裁定，未被 AI 越权定性。
- [ ] 未出现"已修复/AI 自动发现 N 个缺陷"等不实表述。
- [ ] 可对齐 `docs/05-defects/T29_缺陷清单.md` 格式。
