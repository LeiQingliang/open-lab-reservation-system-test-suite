# 提示词 02 · 测试用例设计

> 用 LLM 把功能点/业务规则转化为结构化测试用例草案，覆盖正常、异常与边界。

| 项目 | 内容 |
|------|------|
| **目的** | 基于需求恢复结果，用等价类/边界值/场景法/判定表生成可追溯的用例草表 |
| **适用阶段** | 流水线 ③ 用例设计 |
| **使用者** | 测试工程师（人工裁剪、补漏、校正预期） |

## 输入（占位符）

- `{{功能点与规则清单}}`：提示词 01 的产出
- `{{用例编号前缀}}`：如 FT/VAL/BIZ/API/UT
- `{{用例表字段}}`：默认 编号｜用例名称｜模块｜优先级｜前置条件｜测试步骤｜测试数据｜预期结果｜设计方法
- `{{重点规则}}`：如 预约冲突检测、timeSlot 范围 1–4、日期合法性、状态流转 0→1/2

## 提示词正文

```
你是一名测试用例设计专家。请基于下面的功能点与业务规则，设计测试用例草表。

【功能点与规则】
{{功能点与规则清单}}
【重点关注规则】
{{重点规则}}
【用例编号前缀】{{用例编号前缀}}
【输出表字段】{{用例表字段}}

设计要求：
1. 综合运用：等价类划分、边界值分析、场景法、判定表，并在"设计方法"列标注所用方法。
2. 每条规则至少给出：1 条正常用例 + 1 条异常用例 + 必要的边界用例
   （如 timeSlot 取 0、1、4、5、99；日期取 过去/今天/未来）。
3. 预期结果必须可判定（明确状态码/返回值/数据库变化），不要写"系统正常"这类模糊预期。
4. 对"代码中缺失的规则"，预期写为"系统应当拒绝"，并在备注标注"若实际接受则为缺陷"。
5. 用 Markdown 表格输出，编号连续、可追溯到具体规则。

不要编造功能点之外的用例；不要假设我未提供的接口存在。
```

## 期望输出

- 一张 Markdown 用例表，字段齐全、编号连续；
- 每条用例标注设计方法与可判定的预期；
- 缺失规则对应的用例带"若接受则为缺陷"备注。

## 人工评审要点

- [ ] 预期结果可判定，无"系统正常运行"等模糊措辞。
- [ ] 边界值齐全（timeSlot/日期/空值/查无记录）。
- [ ] 每条用例可追溯到功能点或规则，无凭空用例。
- [ ] 与 `docs/03-test-design/测试用例总表.md` 字段/编号风格一致。
