# 提示词 03 · 单元测试代码生成（JUnit5 + Mockito，不依赖 DB）

> 用 LLM 把 Service 层用例转化为可编译的 JUnit5 + Mockito 单元测试草稿，**不依赖数据库/Redis/Spring 上下文**。

| 项目 | 内容 |
|------|------|
| **目的** | 生成隔离数据访问层（Mapper）的 Service 单元测试，验证业务逻辑的调用/传参/返回处理 |
| **适用阶段** | 流水线 ④ 用例/代码生成 |
| **使用者** | 测试工程师（人工评审 → 本地编译运行确认） |

## 输入（占位符）

- `{{被测Service全名}}`：如 `com.labs.reservation.service.impl.ReservationServiceImpl`
- `{{被测方法签名}}`：粘贴待测方法源码
- `{{依赖Mapper接口}}`：如 `com.labs.reservation.mapper.ReservationMapper`
- `{{domain类字段}}`：如 `Reservation{id,userId,labId,timeSlot,purpose,status}`
- `{{用例清单}}`：提示词 02 中对应该 Service 的用例
- `{{用例编号前缀}}`：如 `UT-RSV-S`

## 提示词正文

```
你是一名 Java 测试开发工程师。请为下面的 Service 生成 JUnit 5 + Mockito 单元测试。

【被测类】{{被测Service全名}}
【被测方法源码】
{{被测方法签名}}
【需 Mock 的依赖】{{依赖Mapper接口}}
【domain 字段】{{domain类字段}}
【要覆盖的用例】
{{用例清单}}

硬性约束（务必遵守）：
1. 仅使用 JUnit 5 (org.junit.jupiter) + Mockito；类上加 @ExtendWith(MockitoExtension.class)。
2. 用 @Mock 隔离 {{依赖Mapper接口}}，用 @InjectMocks 注入被测 Service。
3. 绝不连接数据库 / Redis / 网络 / 真实 Spring 容器；不读系统时钟/随机数，保证确定性。
4. 每个 @Test 用 @DisplayName 标注用例编号（前缀 {{用例编号前缀}}）+ 中文说明。
5. 结构遵循 Arrange–Act–Assert；公共夹具放 @BeforeEach。
6. 断言必须有意义：校验返回值、校验对 Mapper 的调用次数与入参（verify + ArgumentCaptor 视情况）。
   禁止 assertTrue(true)、仅 assertNotNull 之类空断言。
7. 覆盖：正常、失败/异常、边界（如返回 0/1、查无记录返回 null）。
8. 注释用简体中文；代码风格对齐现有测试类（包路径与被测代码一致）。

输出：一个完整可编译的 .java 测试类。生成后另起一段，列出"本测试的可疑点/需人工确认项"。
```

## 期望输出

- 一个完整 `XxxTest.java`（JUnit5+Mockito，确定性、无 DB 依赖）；
- 附带 AI 自评的"可疑点清单"，供人工重点核查。

## 人工评审要点

- [ ] 能通过 `./mvnw -B -pl labs-management -am clean test` 编译运行（在 `system-under-test/back/labs` 下）。
- [ ] 断言有意义、无臆造期望值；`verify` 的调用次数/入参正确。
- [ ] 完全不依赖 DB/Redis/Spring 上下文，结果确定、可重复。
- [ ] `@DisplayName` 编号可追溯；命名/包结构与现有 7 个测试类一致。
- [ ] AI 自评的"可疑点"已逐一确认或修正。
