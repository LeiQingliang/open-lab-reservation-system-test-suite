# 提示词 01 · 系统分析与需求恢复

> 用 LLM 从被测源码出发，归纳系统结构并"逆向恢复"功能需求与业务规则。本项目缺少原始需求文档，故以**源码即需求**为方法。

| 项目 | 内容 |
|------|------|
| **目的** | 让 AI 通读后端源码，输出模块/接口清单、关键控制流、以及反推出的功能点与业务规则 |
| **适用阶段** | 流水线 ① 源码分析 / ② 需求恢复 |
| **使用者** | 测试工程师（人工指定范围并校验结论） |

## 输入（占位符）

- `{{被测模块路径}}`：如 `system-under-test/back/labs/labs-management`
- `{{关注模块}}`：如 预约 reservation、排期 apply、实验室 labs_browse
- `{{真实API列表}}`：见仓库 README / 本目录上级说明
- `{{源码片段}}`：粘贴 Controller / Service / Mapper / domain 关键代码

## 提示词正文

```
你是一名资深软件测试架构师。下面给你一个基于 RuoYi 3.8.9 二次开发的
Spring Boot 2.5 + MyBatis 后端模块的源码，请只依据我提供的代码做分析，
不要臆造代码里不存在的接口、字段或规则。

【被测模块】{{被测模块路径}}
【关注模块】{{关注模块}}
【已知真实 API】
{{真实API列表}}
【源码片段】
{{源码片段}}

请输出：
1. 模块清单：每个模块的 Controller / Service / Mapper / domain 职责一句话概括，
   并标注对应的类名与方法名（必须能在我给的代码里定位到）。
2. 接口清单：HTTP 方法 + 路径 + 入参 + 出参 + 是否需要鉴权（如 @Anonymous）。
3. 关键控制流：列出含分支/循环的核心方法，描述其判定条件与圈复杂度估计。
4. 需求恢复：从代码反推"系统意图实现的功能点"。
5. 业务规则恢复：列出应当存在的业务规则（如预约冲突检测、timeSlot 取值范围、
   日期合法性、状态流转），并对每条标注【代码中已实现】或【代码中缺失/疑似缺陷】。

要求：
- 每条结论后用括号给出来源（类名#方法名），无法定位的不要写。
- 区分"应有规则"与"现状实现"，不要把缺陷当成需求。
- 如信息不足，明确写"需补充源码：xxx"，不要猜。
```

## 期望输出

- 可定位到具体类/方法的模块与接口清单；
- 功能点与业务规则两张清单，规则逐条标注"已实现/缺失"；
- 明确的"信息不足"提示项。

## 人工评审要点

- [ ] 每条结论都能在源码中找到出处，无凭空生成的接口/字段。
- [ ] "缺失规则"的判断与真实缺陷一致（如冲突检测缺失 = 真实 BUG-001 方向）。
- [ ] 未把实现缺陷误写为系统需求。
- [ ] AI 标注的"信息不足"已由人工补料或确认可忽略。
