# 提示词 04 · 接口测试生成（MockMvc standaloneSetup）

> 用 LLM 生成 Controller 层接口测试草稿，采用 MockMvc 的 `standaloneSetup`，**绕过完整 Spring 容器与安全过滤链**，不依赖 DB/Redis。

| 项目 | 内容 |
|------|------|
| **目的** | 对 Controller 的 HTTP 行为（路径、入参绑定、状态码、响应体）做断言 |
| **适用阶段** | 流水线 ④ 用例/代码生成 |
| **使用者** | 测试工程师（人工评审 → 本地编译运行确认） |

## 输入（占位符）

- `{{被测Controller全名}}`：如 `com.labs.reservation.controller.ReservationController`
- `{{Controller源码}}`：粘贴含路由与方法的源码
- `{{依赖Service接口}}`：需 Mock 的 Service
- `{{真实路由}}`：如 `POST /reservation/reservation/insert`、`PUT /reservation/reservation/review`
- `{{用例清单}}`：提示词 02 中对应该 Controller 的接口用例
- `{{用例编号前缀}}`：如 `API-RSV`

## 提示词正文

```
你是一名接口测试工程师。请为下面的 Spring MVC Controller 生成 MockMvc 接口测试。

【被测 Controller】{{被测Controller全名}}
【源码】
{{Controller源码}}
【需 Mock 的 Service】{{依赖Service接口}}
【真实路由】{{真实路由}}
【要覆盖的用例】
{{用例清单}}

硬性约束：
1. 用 MockMvc 的 standaloneSetup 仅装载被测 Controller：
   MockMvcBuilders.standaloneSetup(controller).build()
   —— 不启动完整 Spring 容器、不加载安全过滤链、不连 DB/Redis。
2. 用 Mockito @Mock 被测 Controller 依赖的 Service，并手动注入。
3. 用 mockMvc.perform(...) 构造请求；JSON 体用 ObjectMapper 序列化。
4. 断言 HTTP 状态码 + 响应体关键字段（jsonPath）；对写操作验证 Service 被正确调用。
5. 路径必须与真实路由完全一致（注意本系统的双段路径，如 /reservation/reservation/insert）。
6. 确定性、可重复；注释用简体中文；@DisplayName 标注 {{用例编号前缀}} 编号。

注意诚信：RuoYi 的鉴权由过滤链处理，standaloneSetup 不含过滤链，
因此"未授权返回 401"这类用例不适合用 standaloneSetup 断言——
若用例涉及鉴权/真实错误码（如真实环境返回 500 而非 401 的现象），
请改在端到端/手工层验证，并在输出中明确说明，不要在此伪造断言。

输出：一个完整可编译的 .java 测试类 + "不适合本层验证的用例清单及原因"。
```

## 期望输出

- 一个完整 Controller 测试类（MockMvc standaloneSetup，无 DB 依赖）；
- 一份"应改由 E2E/手工验证的用例清单"（如鉴权错误码）。

## 人工评审要点

- [ ] 使用 `standaloneSetup`，未加载完整容器/安全链/DB。
- [ ] 路由与真实接口完全一致（双段路径无误）。
- [ ] 断言覆盖状态码与关键响应字段，写操作有 `verify`。
- [ ] 鉴权类用例未在此层伪造断言，已正确移交 E2E/手工。
- [ ] 可本地编译运行、确定性稳定。
