# 第2部分：测试计划

## 2.1 测试总体策略

### 2.1.1 策略概述

采用**渐进式测试策略**（Progressive Testing Strategy）：

```
第1阶段: 冒烟测试 (1天)
    ↓
第2阶段: 功能测试+界面测试+输入校验 (7天)
    ↓
第3阶段: 业务规则测试+数据库测试+接口测试 (5天)
    ↓
第4阶段: 白盒测试+单元测试 (4天)
    ↓
第5阶段: 安全测试+性能测试 (2天)
    ↓
第6阶段: 回归测试+缺陷验证 (2天)
```

### 2.1.2 测试方法组合

| 测试方法 | 占比 | 说明 |
|----------|------|------|
| 黑盒测试（功能/界面/业务规则） | 55% | 主要测试方法 |
| 白盒测试（代码路径/单元测试） | 20% | 关键代码 |
| 灰盒测试（数据库/接口） | 15% | 结合代码知识 |
| 手工测试 | 40% | 界面/探索性测试 |
| 自动化测试 | 60% | API/单元/性能 |

### 2.1.3 测试优先级

| 优先级 | 定义 | 模块分配 |
|--------|------|----------|
| P0-最高 | 核心业务流程，失败=系统不可用 | 预约提交、审核通过、认证登录 |
| P1-高 | 重要功能，失败=用户体验严重下降 | 实验室CRUD、排期发布、预约列表查询 |
| P2-中 | 辅助功能，失败=局部影响 | Excel导出、今日排期、用户管理 |
| P3-低 | 非核心功能 | 字典管理、岗位管理、通知公告 |

## 2.8 缺陷管理流程

```
┌─────────┐    ┌──────────┐    ┌───────────┐    ┌──────────┐    ┌──────────┐
│ 发现缺陷 │ → │ 记录缺陷  │ → │ 严重程度  │ → │ 根因分析  │ → │ 制定修复  │
│         │    │ (编号/模块 │    │ 分级      │    │ (代码级)  │    │ 方案      │
└─────────┘    └──────────┘    └───────────┘    └──────────┘    └──────────┘
                                                                    ↓
┌─────────┐    ┌──────────┐    ┌───────────┐    ┌──────────┐    ┌──────────┐
│ 关闭缺陷 │ ← │ 回归测试  │ ← │ 修复验证  │ ← │ 代码修改  │ ← │          │
│         │    │ (通过)    │    │           │    │           │    │          │
└─────────┘    └──────────┘    └───────────┘    └──────────┘    └──────────┘
```

### 2.8.1 缺陷等级定义

| 等级 | 名称 | 定义 | 示例 |
|------|------|------|------|
| 致命 | Critical | 系统无法启动/核心流程完全不可用/数据大范围损坏 | 启动即崩溃；所有预约提交失败 |
| 严重 | Major | 重要模块功能失效/关键业务规则缺失/数据一致性破坏 | 无冲突检测；日期不校验 |
| 一般 | Minor | 局部功能错误但存在替代方案/非核心模块缺陷 | 前端时间显示不一致 |
| 轻微 | Trivial | 界面显示/提示信息等低影响问题 | 提示语不准确 |

### 2.8.2 缺陷生命周期

```
NEW(新建) → OPEN(打开) → FIXED(已修复) → VERIFIED(已验证) → CLOSED(已关闭)
                ↓              ↓                  ↓
             REJECTED(拒绝)  DUPLICATE(重复)   REOPEN(重开)
```

## 2.9 暂停与恢复策略

### 暂停标准
1. 系统无法启动（后端8080端口无响应、前端页面白屏）
2. 数据库崩溃或数据损坏
3. 核心API `/login` 或 `/reservation/reservation/insert` 不可用
4. Redis不可用导致所有认证失败
5. 关键测试环境变更（JDK版本不兼容、操作系统重装）

### 恢复流程
1. 记录暂停时间、原因、影响范围
2. 解决问题并验证（重新执行冒烟测试）
3. 评估是否需要补充测试
4. 更新测试计划，通知项目组
